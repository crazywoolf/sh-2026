# Этап 1 — Ловушки качества данных Meridian

Этот документ фиксирует типичные ошибки, на которых аналитик или AI-агент ошибётся при работе с данными Meridian, если не знает их особенностей. Каждая ловушка доказана реальным SQL к `db/meridian.duckdb` с фактическими числами. Документ — прямой вход для **Critic-агента**: правила ниже используются для проверки и отбраковки ответов, нарушающих семантику витрин.

Условные обозначения: 🔴 — критичная ловушка (ведёт к грубо неверным цифрам), 🟡 — важная (ведёт к нерепрезентативным или неаккуратным выводам).

---

### 🔴 1. orders ≠ financials_monthly

- **Суть:** `orders` и `financials_monthly` — два независимых слоя данных, они НЕ сходятся. Сумма выручки/GMV по завершённым заказам не равна показателям из финансовой витрины (расхождение в разы и на порядок). Это не ошибка данных, а разные сущности: `financials_monthly` — P&L платформы, `orders` — транзакционная лента.
- **Доказательство:**
  `SELECT revenue_gross, gmv FROM financials_monthly WHERE month='2023-12-01';` → revenue_gross = **642 884 746** (~642.9M), gmv = **10 488 811 344** (~10.49B).
  `SELECT sum(gmv) gmv, sum(revenue) rev FROM orders WHERE order_date>='2023-12-01' AND order_date<'2024-01-01' AND status='completed';` → gmv = **2 025 725 180** (~2.03B), rev = **135 905 702** (~135.9M).
  Итог за Dec-2023: revenue_gross 642.9M против orders.rev 135.9M (**≈4.7×**), GMV 10.49B против 2.03B (**≈5.2×**).
- **Правило для Critic-агента:** P&L / выручку / GMV платформы брать ТОЛЬКО из `financials_monthly` (или `unit_economics_monthly`); транзакционную аналитику (заказы, статусы, AOV) — ТОЛЬКО из `orders`. Запрещено агрегировать одно в другое и «сверять» суммы между слоями. Ответ, где выручка платформы посчитана через `SUM(orders.revenue)`, считать ошибочным.

### 🔴 2. Файлы `_params.npz` / `_customers.npz` — не данные

- **Суть:** файлы `data/_params.npz` и `data/_customers.npz` — служебные параметры генератора синтетических данных. В БД они НЕ загружены (отсутствуют среди таблиц) и не являются бизнес-данными.
- **Доказательство:**
  `SHOW TABLES;` → возвращает 8 таблиц: `churn_reasons`, `customer_activity_monthly`, `customers`, `financials_monthly`, `nps_responses`, `orders`, `product_lines`, `unit_economics_monthly`. Файлов `_*.npz` среди них нет.
  `ls data/_*.npz` → `data/_customers.npz`, `data/_params.npz` существуют на диске, но в витринах отсутствуют.
- **Правило для Critic-агента:** игнорировать содержимое `_*.npz`; не включать его в ответы и не ссылаться на него как на источник бизнес-фактов. Источник истины — 8 таблиц БД.

### 🟡 3. Разнородные группы — нельзя усреднять без среза

- **Суть:** усреднение по `segment` / `industry` / `product_line` без разреза маскирует реальную структуру: категории продуктовых линий имеют принципиально разную маржу. «Средняя маржа по всем линиям» — бессмысленный показатель.
- **Доказательство:**
  `SELECT category, count(*) FROM product_lines GROUP BY 1;` → high_margin = **3**, mid_margin = **4**, low_margin = **2**.
  `SELECT pl.category, round(avg(ue.gross_margin_pct),3) gm FROM unit_economics_monthly ue JOIN product_lines pl USING(product_line_id) GROUP BY 1 ORDER BY 2 DESC;` → high_margin = **0.446** (44.6%), mid_margin = **0.288** (28.8%), low_margin = **0.120** (12.0%). Разброс почти 4× между категориями.
- **Правило для Critic-агента:** любой агрегат по неоднородной группе (segment / industry / product_line / category) обязан сопровождаться срезом по подгруппам или явным дисклеймером о разнородности. Единое «среднее» без разреза маркировать как потенциально вводящее в заблуждение.

### 🟡 4. Линия 9 (Консалтинг) — sunset

- **Суть:** продуктовая линия 9 «Консалтинг» имеет статус `sunset` — это закрытая (выводимая из ассортимента) линия. Её включение в активные метрики искажает картину; исключение без оговорки — теряет часть истории.
- **Доказательство:**
  `SELECT product_line_id, name, status FROM product_lines WHERE status='sunset';` → единственная строка: product_line_id = **9**, name = **Консалтинг**, status = **sunset**.
- **Правило для Critic-агента:** при расчётах по продуктовым линиям включение/исключение линии 9 должно быть осознанным и проговорённым в ответе. Ответ, где sunset-линия молча смешана с активными без упоминания её статуса, помечать как неаккуратный.

### 🟡 5. competitor_named пуст у ~83%

- **Суть:** поле `competitor_named` в `churn_reasons` заполнено лишь у меньшинства записей. Выводы о конкретных конкурентах статистически нерепрезентативны.
- **Доказательство:**
  `SELECT count(*) FILTER(WHERE competitor_named IS NULL OR competitor_named='') empty, count(*) total, round(100.0*count(*) FILTER(WHERE competitor_named IS NULL OR competitor_named='')/count(*),1) pct FROM churn_reasons;` → empty = **7335**, total = **8873**, pct = **82.7%** пустых. Заполнено лишь ~17.3% (1538 записей).
- **Правило для Critic-агента:** любой вывод о конкретных конкурентах обязан указывать покрытие (долю заполненных записей ~17%). Ранжирование конкурентов по 17% выборки без дисклеймера о репрезентативности считать ошибочным.

### 🟡 6. Nulls со смыслом — не нули

- **Суть:** NULL в ключевых полях несёт семантику, а не «отсутствие данных». `churn_date IS NULL` означает активного клиента (не уволенного), а не пропуск. `nps_at_churn IS NULL` нельзя трактовать как 0 (это исказит средний NPS вниз).
- **Доказательство:**
  `SELECT count(*) FILTER(WHERE churn_date IS NULL) active_customers FROM customers;` → **16127** активных клиентов (churn_date пуст = активны).
  `SELECT count(*) FILTER(WHERE nps_at_churn IS NULL) null_nps FROM churn_reasons;` → **5144** записей с NULL nps_at_churn.
- **Правило для Critic-агента:** NULL трактовать по семантике поля. `churn_date IS NULL` = «активный клиент», а не «нет данных». `nps_at_churn IS NULL` НЕ включать в расчёт как 0; считать средние с `WHERE ... IS NOT NULL`. Ответ, подставляющий 0 вместо NULL в метрики, отбраковывать.

### 🟡 7. Статусы заказов — выручка только по completed

- **Суть:** в `orders` сосуществуют разные статусы. Выручку нужно считать только по `status='completed'`; cancelled / refunded / disputed представляют отдельные риск-сценарии и анализируются отдельно, а не суммируются в выручку.
- **Доказательство:**
  `SELECT status, count(*) n, round(100.0*count(*)/sum(count(*)) OVER(),1) pct FROM orders GROUP BY 1 ORDER BY 2 DESC;` → completed = **629 348** (**92.4%**), cancelled = **30 179** (**4.4%**), refunded = **14 936** (**2.2%**), disputed = **6842** (**1.0%**).
- **Правило для Critic-агента:** при подсчёте выручки/GMV из `orders` всегда фильтровать `status='completed'` (или явно проговаривать выбранный статус). Суммирование выручки по всем статусам без фильтра считать ошибкой; cancelled/refunded/disputed (вместе ~7.6% заказов) анализировать как отдельные сегменты.

### 🟡 8. Легенда ≠ данные (YoY выручки)

- **Суть:** легенда кейса заявляет «−4% выручки г/г», но по данным витрины фактический YoY выручки иной. Цифры из текста кейса — контекст, а не факт из данных.
- **Доказательство:**
  `SELECT year(month) y, round(100.0*(sum(revenue_net)-lag(sum(revenue_net)) OVER(ORDER BY year(month)))/lag(sum(revenue_net)) OVER(ORDER BY year(month)),1) yoy FROM financials_monthly GROUP BY 1 ORDER BY 1;` → 2024 YoY = **−11.5%**, 2025 YoY = **−5.3%** (2023 — базовый год, YoY = NULL). Ни одно значение не равно «−4%» из легенды.
- **Правило для Critic-агента:** отвечать ПО ДАННЫМ. Цифры из легенды кейса («−4%» и т.п.) помечать как контекст/заявление кейса, а не как факт из витрины. Ответ, который выдаёт легендарную цифру как посчитанный из БД факт, отбраковывать; проверять метрики против реального SQL.
