# Описание кейса «Meridian» — AI South Hub 2026

Источник: https://dashboard.aisouthhack.ru/case

## Бизнес-ситуация

**Meridian** — B2B-маркетплейс услуг для компаний среднего сегмента (mid-market). Проблемы:
- Выручка снижается: **−4% год к году**.
- Отток клиентов: **23% в год**.
- Клиенты уходят к специализированным AI-инструментам вместо использования Meridian как посредника-маркетплейса.
- Аналитический отдел работает по медленной тикетной модели: бизнес-вопрос → дашборд → ответ за **5–10 дней**.

## Главный вызов

Построить **мультиагентную аналитическую систему, работающую как data-аналитик для совета директоров** — без традиционных задержек. Система принимает бизнес-вопросы на естественном языке и выдаёт обоснованные, подкреплённые данными ответы с явными допущениями и честными заявлениями об ограничениях данных.

## Три режима работы

1. **Conversational BI Assistant** — руководители задают вопросы в свободной форме и получают краткие ответы с поддерживающими визуалами и удержанием контекста между уточняющими вопросами.

2. **Ad-hoc Research** — система самостоятельно декомпозирует сложные вопросы и проводит мини-исследования со сравнительным анализом.

3. **Scheduled Reports** — автоматические дашборды компилируют метрики здоровья бизнеса по заранее заданному расписанию и рассылке.

## Требуемая архитектура: четыре агента

### Data Extraction Agent (Extractor)
Интерпретирует бизнес-запросы относительно схемы данных, формулирует SQL, распознаёт сценарии недостаточности данных.

### Analyst Agent
Выводит инсайты на языке бизнеса из датасетов, обнаруживает паттерны и аномалии, переводит числовые находки в стратегические следствия.

### Critic Agent
Валидирует интерпретацию вопроса, уместность среза данных, обоснованность выводов; флагает галлюцинации и подмены. **От этого компонента сильно зависит итоговый балл** — нужны конкретные проверки под выявленные категории ошибок, а не общий мониторинг.

### Visualization Agent
Выбирает подходящие типы графиков; понимает разницу между показом динамики, сравнением категорий, выявлением аномалий и отображением структуры.

## Критерии оценки (с весами)

| Критерий | Вес |
|---|---|
| Точность ответов | 30% |
| Boundary discipline (корректное заявление о недостаточности данных) | 15% |
| Качество бизнес-интерпретации | 25% |
| Архитектура агентов | 20% |
| Визуализация и UX | 10% |

Итоговая формула: **45% — автоматическая оценка LLM-судьёй по живому API, 55% — экспертная оценка жюри.**

## Культурное требование

> «Meridian — компания с инженерной культурой. Здесь "мне кажется" не работает. Здесь работает "вот цифры, вот метод подсчёта, вот допущения".»

Система должна соответствовать этой строгости:
- не усреднять по разнородным группам;
- явно проговаривать допущения;
- различать уверенные выводы и ограничения данных.

## Поставки (deliverables)

- Рабочая система, развёрнутая в **Yandex Cloud**.
- Документация архитектуры (схема + 1–2 страницы).
- Транскрипт взаимодействия агентов на одном тест-кейсе.
- Защита: **7-минутная презентация + 5 минут Q&A**.
