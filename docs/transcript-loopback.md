# Транскрипт работы агентов Meridian

**Тест-кейс (вопрос):** Какое отношение LTV/CAC по сегментам? Где привлечение убыточно?

Документ показывает, **как система пришла к выводу** — по шагам конвейера агентов. Каждый этап снабжён пояснением «что делает агент» (роль) и «что произошло здесь» (интерпретация конкретного шага).

```
Вопрос → Planner → Extractor → Analyst → Critic → Visualization → Ответ
```
> Сквозная гарантия: система **никогда не возвращает 500** и **не выдумывает данные** — если их нет, честно помечает `insufficient_data`.

---

## 1. Planner — классификация и декомпозиция
> **Роль агента:** понимает вопрос и решает РЕЖИМ обработки: `bi` (один срез данных), `research` (декомпозиция на под-вопросы) или `insufficient` (нужного поля нет в витрине). Для сложных вопросов разбивает их на самодостаточные под-вопросы.

```json
{
  "mode": "bi",
  "reasoning": "Вопрос требует предоставления данных по отношению LTV к CAC в разбивке по сегментам, что можно сделать с помощью данных из таблицы unit_economics_monthly. Это один срез данных, не требуется собирать информацию из нескольких независимых источников.",
  "sub_questions": [
    "Какое отношение LTV/CAC по сегментам? Где привлечение убыточно?"
  ]
}
```

**→ Что произошло:** выбран режим **`bi`** (диалоговый BI — отвечается одним срезом данных); вопрос разложен на **1** под-вопрос(а). Дальше каждый под-вопрос проходит Extractor → Analyst → Critic.

## 2. Extractor — извлечение данных (вопрос → SQL → DuckDB)
> **Роль агента:** превращает под-вопрос в данные. Берёт готовую **проверенную метрику** из библиотеки, либо пишет безопасный `SELECT` (read-only, guard-rails: только чтение, белый список таблиц). Сам определяет, достаточно ли данных.

- **под-вопрос:** Какое отношение LTV/CAC по сегментам? Где привлечение убыточно?
- **подход:** готовая метрика `ltv_cac_by_segment`
- **данных достаточно:** да · **получено строк:** 3
- **SQL (исполнен на DuckDB read-only):**
```sql
SELECT segment, round(avg(ltv_12m)/avg(cac),2) AS ltv_cac
          FROM unit_economics_monthly WHERE month='2025-12-01' GROUP BY 1 ORDER BY 2 DESC
```
- **первые строки результата:**
```json

  {
    "segment": "Large",
    "ltv_cac": 3.03
  },
  {
    "segment": "Mid",
    "ltv_cac": 1.68
  },
  {
    "segment": "SMB",
    "ltv_cac": 0.59
  }
]
```

**→ Что произошло:** данные получены (3 строк) — Extractor передаёт их Analyst для интерпретации.

## 4. Critic — валидация и петля доработки (ключевой компонент оценки)
> **Роль агента:** независимо проверяет ответ по чек-листу ловушек (ложный отказ, смешение таблиц, усреднение разнородного, сверка чисел со строками, масштаб, стратегический смысл). При несоответствии возвращает работу назад (`revise`) с КОНКРЕТНЫМ указанием — до 2 раз; система применяет указание и пересобирает ответ. Принцип **keep-best**: ревизию принимаем, только если она НЕ потеряла данные — иначе оставляем исходный верный ответ (защита от уверенно-неправильного критика).

### Итерация 1 — вердикт `revise` (8/9 проверок пройдено)
- ❌ **есть стратегический смысл в ответе:** Стратегический смысл присутствует, но рекомендация могла бы быть более развёрнутой и содержательной.

> 🔁 **Доработка 1** (цель: `analyst`). Указание ревизора:
> «Расширить стратегические рекомендации, предложив конкретные действия для оптимизации затрат на привлечение в сегменте SMB и анализа причин низкого LTV. Например, рассмотреть возможность анализа воронки привлечения, оценки эффективности каналов привлечения, анализа поведения клиентов в сегменте SMB и т. д.»

- **Analyst** переписал вывод с учётом указания.

### Итерация 2 — вердикт `revise` (8/9 проверок пройдено)
- ❌ **Усреднение разнородных групп без оговорки:** В запросе используется усреднение показателей LTV и CAC по месяцам (unit_economics_monthly), что может дать искажённое представление о соотношении LTV/CAC в сегментах. Следует использовать данные, которые отражают более долгосрочную динамику или совокупные значения для каждого сегмента.

> 🔁 **Доработка 2** (цель: `extractor`). Указание ревизора:
> «Использовать данные, которые отражают более долгосрочную динамику или совокупные значения LTV и CAC для каждого сегмента, вместо усреднения по месяцам в unit_economics_monthly.»

- **Extractor** пересобрал запрос с учётом замечания: подход метрика `ltv_cac_by_segment`, строк 3, данные на месте ✅.
- **Analyst** переписал вывод с учётом указания.

### Итерация 3 — вердикт `revise` (8/9 проверок пройдено)
- ❌ **Наличие стратегического смысла:** Стратегический анализ поверхностный, нет конкретных предложений по оптимизации CAC или повышения LTV, кроме общего совета провести анализ.

**→ Лимит ревизий (2) исчерпан.** По принципу **keep-best** система отдаёт лучший непадший вариант (данные на месте, числа верны), а замечание ревизора сохраняется как оговорка. Здесь замечание спорное — каноничная метрика берёт срез одного месяца, а не усредняет панель, — поэтому исходный верный ответ не деградирует.
