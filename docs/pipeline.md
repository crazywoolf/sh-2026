# API-контракт (по нему тестирует LLM-судья)

## Endpoint
POST на один из путей (проверяются по порядку):
`/api/chat` → `/api/v1/chat` → `/chat` → `/api/ask` → `/api/query`

## Формат запроса
```json
{"message": "Вопрос на естественном языке"}
{"query": "Альтернативное поле"}
{"messages": [{"role": "user", "content": "..."}]}
```
Для диалога: добавить `"session_id": "abc123"`.

## Формат ответа
Минимум:
```json
{"response": "Вывод на языке бизнеса"}
```
Валидные имена поля ответа: `response`, `answer`, `message`, `content`, `text`.

Рекомендованный расширенный (с трассировкой):
```json
{
  "response": "Бизнес-вывод",
  "assumptions": ["условные допущения"],
  "trace": [
    {"agent": "extractor", "sql": "SELECT...", "rows": 120},
    {"agent": "analyst", "note": "наблюдение"},
    {"agent": "critic", "verdict": "approved"}
  ],
  "chart": {"type": "line", "x": "quarter", "y": "revenue"},
  "insufficient_data": false,
  "session_id": "abc123"
}
```
На вопрос без ответа: `"insufficient_data": true`, **не выдумывать.**

# Автоматические проверки качества

**1. Соответствие контракту**
- Endpoint принимает POST с JSON.
- Ответы — осмысленный русский текст (UTF-8, ≥10 символов).
- Вне-скоуп вопросы → «недостаточно данных».

**2. Обработка ошибок (никаких 500)**
- Пустое тело → 400; кривой JSON → 400; нет поля вопроса → 400/422; несуществующий путь → 404.
- Ошибки — JSON-полем `error` или `detail`.

**3. Производительность**
- Простой вопрос < 5 мин; аналитика < 10 мин.
- 3 параллельных запроса — все получают ответ.
- Мульти-вопросные сессии — сохранение контекста.

**4. Робастность под стрессом (20% балла)**
- Длинные входы (5000+ символов) не роняют систему.
- Спецсимволы, эмодзи, SQL/HTML в вопросе — без 500.
- Попытки prompt injection — поведение не меняется.
- Несовпадение типов (null, числа вместо строк) — корректная ошибка.

**5. Бонусы**
- `GET /health → 200`.
- `GET /docs` или `/openapi.json`.
- CORS-заголовки.
- Streaming (SSE) для диалога: `text/event-stream`.

# Минимальный FastAPI-сервер

```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

class ChatRequest(BaseModel):
    message: str
    session_id: str | None = None

@app.post("/api/chat")
async def chat(req: ChatRequest):
    if not req.message.strip():
        raise HTTPException(status_code=400, detail="empty message")

    result = await run_agents(req.message, req.session_id)

    return {
        "response": result.answer,
        "assumptions": result.assumptions,
        "trace": result.trace,
        "insufficient_data": result.insufficient_data,
        "session_id": req.session_id,
    }

@app.get("/health")
async def health():
    return {"status": "ok"}
```

# Стратегические принципы
1. **Сначала Planning Mode** — сформулировать, как должен выглядеть продукт, до генерации кода (экономит токены, предотвращает провалы интеграции).
2. **Legend First** — первый час разбирать витрину как аналитик; экономит дни отладки.
3. **Critic — ядро** — балл сильно зависит от самодиагностики; делать конкретные проверки под категории ошибок, а не общий мониторинг.
4. **Универсальная архитектура** — проектировать под 3 режима сразу.
5. **Готовность к защите** — жюри тестит систему напрямую; готовить и примеры отличной работы, и честные отказы.

**Фундаментальное правило:** никогда не возвращать 500. Любой ввод — валидный, кривой, пустой, огромный, провокационный — требует осмысленного ответа, честного «недостаточно данных» или корректного кода ошибки (400/422).
