# Дизайн: Этап 5 — Дашборд, UX и три режима

**Дата:** 2026-06-11
**Хакатон:** AI South Hub 2026, кейс «Meridian»
**Этап пайплайна:** Stage 5 — Dashboard & UX (Checkpoint 2, вечер 12 июня)

## Контекст и цель

Бэкенд готов и проверен вживую (Node/Fastify, `/api/chat`, оркестратор 4 агентов, YandexGPT; этап 4 — flag 18/18, 0 wrong). Нужно дать системе «лицо» для защиты и реализовать **три операционных режима кейса**: Диалоговый BI, Ad-hoc исследование, Отчёты по расписанию. Плюс недостающие бэкенд-части: удержание контекста сессии и реальный планировщик отчётов. Поставка этапа — этот дизайн; код — следующим шагом.

## Ключевые решения (согласовано с пользователем)

1. **Фронтенд:** статичный SPA (`web/index.html` + `app.js` + `styles.css`), раздаётся тем же Fastify. Без сборки/фреймворка.
2. **Дизайн-система:** Gravity UI (открытая DS Яндекса) — через **дизайн-токены** (CSS-переменные `--g-*`: палитра, типографика, радиусы). Если CDN-стили не подхватятся офлайн — продублировать токены своим CSS.
3. **Графики:** Chart.js с CDN, кормится `chart`-спекой из ответа. Сквозные во всех режимах.
4. **Реальный серверный планировщик** отчётов (`node-cron`, чистый JS) — не симуляция.
5. **Доставка отчётов pluggable:** всегда во внутренний инбокс; опционально webhook; **Telegram — stretch-цель** (если останется время).

## Три режима (канон кейса)

**① Диалоговый BI** — чат: NL-вопрос → короткий ответ с обоснованием и графиком; держит контекст диалога (`session_id` + SessionStore). Backend: `bi`-режим планировщика.

**② Ad-hoc исследование** — сложный вопрос с гипотезой → система сама декомпозирует на под-вопросы и проводит мини-исследование, затем синтез. Backend: `research`-режим оркестратора (уже реализован). UI показывает **декомпозицию** (поставленные под-вопросы), находку по каждому и итоговый вывод. Чтобы режим надёжно демонстрировал исследование, запрос из вкладки ② передаёт опциональный флаг `prefer_research: true` — Planner трактует его как сильную подсказку выбрать `research` (но всё равно может вернуть `insufficient` для невозможных вопросов). Без флага (вкладка ①) — обычная авто-классификация.

**③ Отчёты по расписанию** — система сама по расписанию собирает **дашборд здоровья компании** (пресет-метрики → карточки + графики, подсветка тревог) и «отправляет». UI показывает расписание, кнопку «Собрать сейчас» и ленту доставленных отчётов.

## Бэкенд-добавки

### Контекст сессий (`src/session/store.ts`)
- In-memory `SessionStore`: `get(id) → recent turns[]`, `append(id, question, answer)`; хранит последние K=5 пар на `session_id`.
- Сервер перед вызовом пайплайна достаёт контекст и передаёт в `runPipeline(agents, query, context)`; после — `append`.
- Оркестратор прокидывает компактную историю в **Planner** (разрешение ссылок «а по сегментам?») и **Extractor**.

### Пресеты и отчёт (`src/presets.ts`, `src/report.ts`)
- `presets.ts`: фикс-набор «вопросов здоровья» (выручка YoY, структура GMV, отток + AI-угроза, LTV/CAC по сегментам, NPS, вовлечённость).
- `report.ts`: `compileReport()` — прогоняет пресеты через пайплайн, собирает `{generatedAt, items: [{question, response, chart, insufficient_data}]}`.

### Планировщик и доставка (`src/scheduler.ts`, `src/delivery.ts`)
- `scheduler.ts`: `node-cron` по выражению из env `SCHEDULE_CRON` (прод — `0 9 * * 1` «пн 9:00»; демо — частый интервал). На срабатывание: `compileReport()` → `deliver(report)`.
- `delivery.ts`:
  - инбокс (in-memory лента доставленных отчётов) — всегда;
  - опц. webhook: POST JSON, если задан `REPORT_WEBHOOK`;
  - опц. **Telegram (stretch)**: POST в `https://api.telegram.org/bot<token>/sendMessage` c `chat_id`+текстом, если заданы `TELEGRAM_BOT_TOKEN`+`TELEGRAM_CHAT_ID`.

## API (расширение)

- `GET /` → `web/index.html` (статика из `web/`).
- `POST /api/chat` (есть) — теперь принимает `session_id`, использует контекст; ответ дополнен полями.
- `POST /api/report` — собрать отчёт сейчас → вернуть + положить в инбокс.
- `GET /api/reports` — лента доставленных отчётов (инбокс).
- `GET /api/presets` — список пресет-вопросов (для дашборда).

### Расширение контракта (не ломает судью)
`FinalResponse` дополняется **опциональными** полями:
```ts
plan?: { mode: "bi" | "research" | "insufficient"; sub_questions: string[] };
sub_answers?: string[];   // по под-вопросам research — для UI режима ②
```
Обязательные поля контракта судьи (`response, chart, trace, assumptions, insufficient_data, session_id`) остаются; лишние поля разрешены спецификацией.

## UX и обработка ошибок

- `insufficient_data: true` → честная плашка «Данных недостаточно» (не пустой экран) — визуализирует boundary discipline.
- Лоадеры на время ответа (5–10с); таймаут с понятным сообщением; никаких 500 на фронт не доходит (бэкенд гарантирует).
- Карточки-«тревоги» на дашборде подсвечиваются (отрицательная динамика выручки, SMB LTV/CAC < 1).
- Секреты только в серверном `.env`; фронт ходит лишь в свой бэкенд.

## Вне scope

- Аутентификация/мультипользовательность дашборда.
- Персистентное хранилище (инбокс/сессии — in-memory; перезапуск очищает).
- Полноценная вёрстка под мобильные (фокус — десктоп защиты).
- Реальная почтовая рассылка (вместо неё — инбокс + опц. webhook/Telegram).

## Критерий готовности

- SPA на Gravity UI открывается на `/`, три режима работают вживую через бэкенд.
- Диалог удерживает контекст (уточняющий вопрос разрешается по истории).
- Исследование показывает декомпозицию + синтез.
- Планировщик реально срабатывает по расписанию, собирает отчёт и кладёт в инбокс (виден в UI); webhook/Telegram — при наличии env.
- Контракт судьи не сломан; юнит-тесты новых модулей зелёные, typecheck чист.
