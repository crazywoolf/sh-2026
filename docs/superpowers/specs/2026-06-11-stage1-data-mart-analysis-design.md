# Дизайн: Этап 1 — Витрина Meridian в DuckDB + аналитические документы

**Дата:** 2026-06-11 (ред. — переход с «только документы» на «БД + документы»)
**Хакатон:** AI South Hub 2026, кейс «Meridian»
**Этап пайплайна:** Stage 1 — Data Mart Analysis

## Контекст и цель

Перед постройкой мультиагентной системы нужно (а) загрузить витрину Meridian в **запрашиваемую базу данных**, чтобы агенты (прежде всего Extractor на этапе 3) и мы выполняли реальный SQL, и (б) задокументировать понимание данных так, как это сделал бы аналитик: что считаемо, где ловушки, на какие вопросы система отвечает и где честно говорит «данных недостаточно».

Всё на реальных данных из 8 CSV в [`data/`](../../../data/).

## Решения (согласовано с пользователем)

- **Хранилище — DuckDB** (файловая аналитическая БД, ноль настройки, читает CSV напрямую, быстрая на агрегациях по сотням тысяч строк). Один файл `db/meridian.duckdb`.
- **Сохраняем ключевые аналитические документы** (карта, метрики, ловушки, банк вопросов) — но теперь они **подкреплены реальным SQL** к DuckDB, а не awk. Каждый артефакт становится проверенной библиотекой запросов для Extractor-агента.
- **Язык документов — русский.**
- **Каталог метрик — рабочий набор** (~18 ключевых метрик), не исчерпывающий.
- **Банк вопросов генерируем сами**, с явными вопросами-ловушками (insufficient data).

## Окружение

- `brew` доступен → DuckDB ставится CLI-бинарником: `brew install duckdb`.
- `python3` сломан (нет CommandLineTools) → загрузчик делаем **на DuckDB SQL**, запускаемом через CLI, без Python.

## Структура поставки

### База данных и загрузчик
- `db/load.sql` — идемпотентный загрузчик: `CREATE OR REPLACE TABLE <t> AS SELECT ... FROM read_csv_auto('data/<t>.csv', header=true)`, с явными типами для дат (DATE) и id (INTEGER/BIGINT), booleans для флагов. 8 таблиц: customers, orders, product_lines, nps_responses, customer_activity_monthly, churn_reasons, financials_monthly, unit_economics_monthly. Файлы `_*.npz` игнорируются.
- `db/build.sh` — сборка: `duckdb db/meridian.duckdb < db/load.sql` + вывод row counts для верификации. Идемпотентно (CREATE OR REPLACE).
- `db/meridian.duckdb` — собранный артефакт. В git **не коммитим** (бинарь, пересобирается из CSV) — добавить в `.gitignore`.

### Документы в `docs/stage1/`
- **`00-database.md`** (NEW) — как собрать (`bash db/build.sh`) и запросить БД; список 8 таблиц с типами; верифицированные row counts; 3–5 примеров запросов.
- **`01-data-map.md`** (✓ готов) — ER-диаграмма + гранулярность. Дополнить короткой заметкой «как запросить БД» со ссылкой на `00-database.md`.
- **`02-metrics-catalog.md`** — ~18 ключевых метрик; у каждой: определение, формула, **рабочий SQL** (проверен на DuckDB), источник, гранулярность, наблюдение. Раздел «НЕ считаемо» (≥4 пункта).
- **`03-data-quality-traps.md`** — ловушки; у каждой: суть, **SQL-доказательство** (проверено), правило для Critic-агента. Обязательны: orders≠financials, `_*.npz`, разнородные группы, sunset-линия, nulls, статусы заказов, расхождение легенды «−4%» с данными.
- **`04-question-bank.md`** — ≥16 вопросов; у отвечаемых — набросок SQL; ≥4 вопроса-ловушки (insufficient_data), опирающихся на «НЕ считаемо»/ловушки.

## Типы данных (ориентир для load.sql)

- даты: signup_date, churn_date, order_date, response_date, launch_date, month → `DATE` (пустые → NULL).
- id: customer_id, order_id, product_line_id, response_id → `INTEGER`; крупные счётчики ок `BIGINT`.
- денежные/доли: gmv, revenue, cac, ltv_12m, take_rate, gross_margin_pct и пр. → `DOUBLE`.
- флаги: interview_completed → `BOOLEAN`.
- остальное (segment, industry, status, category, *_reason, *_tag …) → `VARCHAR`.
- read_csv_auto обычно выводит это корректно; явные типы задаём минимум для дат и ключей.
