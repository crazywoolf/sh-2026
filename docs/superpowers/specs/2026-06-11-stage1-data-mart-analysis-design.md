# Дизайн: Этап 1 — Анализ витрины данных Meridian

**Дата:** 2026-06-11
**Хакатон:** AI South Hub 2026, кейс «Meridian»
**Этап пайплайна:** Stage 1 — Data Mart Analysis (до написания кода)

## Контекст и цель

Перед постройкой мультиагентной аналитической системы нужно понять данные так, как это сделал бы аналитик. По итогу этапа любой член команды и любой будущий агент должны понимать:
- **что** в витрине есть и как таблицы связаны;
- что из данных **считаемо** (и чего нет);
- где **ловушки** качества данных;
- на какие **вопросы** система должна отвечать — и где честно говорить «данных недостаточно».

Всё на реальных цифрах из CSV в [`data/`](../../../data/). **Кода на этом этапе не пишем** — только аналитические документы. Опора — уже собранный [`docs/data-dictionary.md`](../../data-dictionary.md).

## Решения (согласовано с пользователем)

- **Только документы**, без DuckDB/ноутбуков/кода.
- **Язык документов — русский** (как кейс и ответы целевой системы).
- **Каталог метрик — рабочий набор** (~15–20 ключевых метрик под легенду кейса), а не исчерпывающий перечень.
- **Банк вопросов генерируем сами** на основе легенды кейса и данных, с явной пометкой вопросов-ловушек («insufficient data»).

## Структура поставки

Четыре коротких сфокусированных документа в `docs/stage1/`:

### ① `01-data-map.md` — карта витрины
- ER-диаграмма (mermaid): 8 таблиц + ключи связи (`customer_id`, `product_line_id`).
- Для каждой таблицы: гранулярность (что = одна строка), период, объём строк, что таблицей можно измерить.
- Явно зафиксировать уровни агрегации: `financials_monthly` — платформа; `unit_economics_monthly` — сегмент×линия×месяц; `orders`/`customer_activity_monthly`/`nps_responses` — транзакционный/событийный уровень; `customers`/`product_lines` — справочники; `churn_reasons` — exit-интервью.

### ② `02-metrics-catalog.md` — каталог метрик (рабочий набор)
~15–20 ключевых метрик. Для каждой: определение, **формула**, исходные таблицы/колонки, гранулярность, на какой вопрос отвечает.
Тематические блоки:
- **P&L:** выручка (gross/net), GMV, take rate, EBITDA, динамика −4% г/г.
- **Юнит-экономика:** CAC, LTV 12m, payback, gross margin, LTV/CAC — по сегменту×линии.
- **Отток:** churn rate, retention, причины оттока, доля `ai_alternative`.
- **Вовлечённость:** распределение activity-статусов (active/churning/dormant), прокси DAU/логины.
- **NPS:** NPS-score, доли промоутеров/детракторов по линиям, теги комментариев.
- **Угроза AI:** доля тегов `ai_competitor` (NPS) и `ai_alternative` (отток), named competitors.

Отдельный раздел **«НЕ считаемо»** — чего в данных нет (заранее очерчивает границы → boundary discipline). Примеры: себестоимость на уровне заказа, маркетинговые расходы по каналам, выручка по городам с причинно-следственной атрибуцией и т.п. — уточнить по факту.

### ③ `03-data-quality-traps.md` — качество данных и ловушки
Для каждой ловушки: суть + «как обходим» (прямой вход для проверок Critic-агента).
Известные на момент дизайна:
- 🔴 **`orders` ≠ `financials_monthly`** — не сходятся ни по выручке, ни по масштабу GMV (Dec 2023: financials revenue ≈ 642.9M против суммы completed-orders ≈ 135.9M; GMV различается ~4×). Это два независимых синтетических слоя — нельзя агрегировать orders в financials и наоборот.
- 🔴 **`_params.npz` / `_customers.npz`** — внутренние параметры генератора данных (`margin_by_line`, `take_rate_base`, per-customer массивы). **Не использовать** — это утечка, а не бизнес-данные.
- Разнородные группы: нельзя усреднять по segment/industry/product_line без явного среза (разная маржинальность low/mid/high_margin).
- Линия 9 (Консалтинг) — `sunset`: включать/исключать осознанно.
- `competitor_named` пуст у ~83% строк оттока — выводы о конкретных конкурентах с оговоркой.
- Nulls: `churn_date` (16 127 активных), `nps_at_churn` (5 144 пустых).
- Статусы заказов: выручку считать по `completed`; cancelled/refunded/disputed — отдельно.

### ④ `04-question-bank.md` — банк вопросов (15–20)
Таблица с колонками: **вопрос · категория** (BI/ad-hoc research/scheduled report) **· сложность · таблицы+метрики для ответа · набросок подхода** (прозой, без кода) **· флаг** answerable / insufficient_data.
Состав: ~70% отвечаемых (падение выручки, отток 23%, уход к AI-конкурентам, юнит-экономика, NPS) + ~30% вопросов-ловушек, где правильный ответ — честное «данных недостаточно» (опираясь на раздел «НЕ считаемо» из ②). Этот банк напрямую питает тесты этапа 4 и boundary discipline.

## Зависимости
- Вход: 8 CSV в `data/`, `docs/data-dictionary.md`, `docs/case.md`.
- Выход: 4 markdown-документа в `docs/stage1/`.
- Используется на: этапе 2 (контракты агентов — что Extractor может извлечь), этапе 3 (первый сквозной вопрос), этапе 4 (тестовый набор и проверки Critic).

## Вне scope
- Любой код, загрузчики, DuckDB, ноутбуки.
- Сама реализация агентов.
- Исчерпывающий перечень всех теоретически считаемых метрик.

## Критерий готовности
Все 4 документа написаны на русском, на реальных цифрах из CSV; ловушки `orders≠financials` и `.npz` зафиксированы; банк содержит ≥15 вопросов с не менее чем 4 вопросами-ловушками.
