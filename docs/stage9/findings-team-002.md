## №1 — NPS по сегментам
**Вопрос:** «Какой NPS по каждому сегменту (Large, Mid, SMB)?»
**Ответ:** «Средний NPS 2025 = 33,48… разбивка по сегментам Large, Mid, SMB **недоступна в текущем срезе**…» (далее верно про survivorship).
**Что не так:** Ложный отказ — NPS по сегментам считается (опросы + сегмент клиента). Система выдала агрегат и заявила, что сегментный разрез недоступен.
**Как должно быть:** Large **45.3**, Mid **23.5**, SMB **22.1** (большой разрыв).
**Проверка:** `SELECT c.segment, round(100.0*(count(*) FILTER(WHERE n.score>=9)-count(*) FILTER(WHERE n.score<=6))/count(*),1) FROM nps_responses n JOIN customers c USING(customer_id) GROUP BY 1;`

## №2 — месяц с минимальным оттоком 2025
**Вопрос:** «В каком месяце 2025 года был самый низкий отток клиентов?»
**Ответ:** «Формальный отток в декабре 2025 = 25, −82,3% к ноябрю — **минимум за год**…»
**Что не так:** Уверенный вывод на цензурированных данных. Декабрь 2025 — последний месяц витрины, счётчик ушедших за него обрезан (ноя 141 → дек 25). Резкое падение — артефакт неполноты, а не реальный минимум. Система сама показывает обрыв (−82,3%), но всё равно называет декабрь «минимумом за год», не пометив цензуринг.
**Как должно быть:** Декабрь нельзя называть минимумом — данные неполны; сравнивать только закрытые месяцы (минимум среди них — ноябрь 141), декабрь пометить как обрезанный.
**Проверка:** `SELECT strftime(churn_date,'%Y-%m') m, count(*) FROM customers WHERE year(churn_date)=2025 GROUP BY 1 ORDER BY 1;` → …ноя 141 / дек 25.

## №7 — доля детракторов/промоутеров
**Вопрос:** «Какая доля детракторов и промоутеров в NPS?»
**Ответ:** «NPS = 34,0 → ≈ 67 % промоутеров и ≈ 33 % детракторов.»
**Что не так:** Неверный вывод: из NPS=34 нельзя получить 67/33 — это игнорирует нейтралов (passives). За 2025 реально промоутеры 47%, нейтралы 40%, детракторы 13% (47−13=34).
**Как должно быть:** 2025: промоутеры 47%, нейтралы 40%, детракторы 13%.
**Проверка:** `SELECT round(100.0*count(*) FILTER(WHERE score>=9)/count(*),1), round(100.0*count(*) FILTER(WHERE score BETWEEN 7 AND 8)/count(*),1), round(100.0*count(*) FILTER(WHERE score<=6)/count(*),1) FROM nps_responses WHERE year(response_date)=2025;` → 47 / 40 / 13

## №12 — клиенты по топ-5 городам

## №13 — доля клиентов по сегментам
**Вопрос:** «Какая доля клиентов приходится на каждый сегмент в процентах?»
**Ответ:** «Mid ≈ 39,6 %, SMB ≈ 35,2 %, Large ≈ 25,2 %.»
**Что не так:** Неверный факт — это доли **GMV** по сегментам, а не доли клиентов. По числу клиентов структура обратная: доминирует SMB.
**Как должно быть:** SMB **62.6%**, Mid **29.5%**, Large **7.9%**.
**Проверка:** `SELECT segment, round(100.0*count(*)/sum(count(*)) OVER(),1) FROM customers GROUP BY 1;` → SMB 62.6 / Mid 29.5 / Large 7.9

## №14 — LTV/CAC по сегментам
**Вопрос:** «Какой LTV/CAC по сегментам Large, Mid, SMB?»
**Ответ:** «CAC в 2025: Large 57 254 ₽, Mid 16 910 ₽, SMB 7 589 ₽.»
**Что не так:** Ответ не на вопрос — выдан только CAC, а само отношение LTV/CAC (то, что спросили) не дано.
**Как должно быть:** LTV/CAC (последний месяц): Large 3.03, Mid 1.68, SMB 0.59 (SMB убыточен).
**Проверка:** `SELECT segment, round(avg(ltv_12m)/avg(cac),2) FROM unit_economics_monthly WHERE month=(SELECT max(month) FROM unit_economics_monthly) GROUP BY 1;`

## №15 — средний чек по сегментам
**Вопрос:** «Какой средний чек по сегментам?»
**Ответ:** «Blended медианный чек 86 790 ₽… внутри каждой из 8 продуктовых линий медиана растёт…»
**Что не так:** Пропущен запрошенный срез — дан общий чек и разбивка по продуктовым ЛИНИЯМ, а не по СЕГМЕНТАМ (Large/Mid/SMB).
**Как должно быть:** Средний чек по сегментам считается через `orders ⋈ customers.segment`.
**Проверка:** `SELECT c.segment, round(avg(o.gmv),0) FROM orders o JOIN customers c USING(customer_id) GROUP BY 1;`
