## Находка №1 — team-009

**Команда:** team-009 (`https://team-009.aisouthhack.ru`, `POST /api/chat`)

**Вопрос:** «Назови главную причину оттока клиентов.» (повторно воспроизведено на «Отток клиентов снизился. Значит ли это, что база стала стабильнее?»)

**Ответ системы:**
> «Главная причина оттока клиентов — высокая цена, на которую приходится 24,6% случаев… Наиболее уязвимый сегмент — SMB, где уровень оттока составляет 36,3%, причём **64% уходов в этом сегменте связаны с переходом на AI-альтернативы**.»

**Что не так:** Утверждение «64% уходов в SMB связаны с AI-альтернативами» не подтверждается данными — это выдумка (вывод без опоры на данные). Доля причины `ai_alternative` среди ушедших SMB-клиентов = **14.7%**, а не 64%. Главная причина в SMB — цена (24.5%). Ошибка устойчивая: повторилась в двух разных ответах.

**Как должно быть:** Распределение причин оттока в сегменте SMB (по витрине):
цена 24.5% · нет потребности 19.1% · качество 16.6% · консолидация 15.9% · **AI-альтернативы 14.7%** · прочее 9.3%.
То есть AI-альтернативы — лишь 5-я по частоте причина (~1/7 уходов), а не 64%.

**Проверка (DuckDB):**
```sql
SELECT cr.primary_reason, round(100.0*count(*)/sum(count(*)) OVER(),1) pct
FROM churn_reasons cr JOIN customers c USING(customer_id)
WHERE c.segment='SMB' GROUP BY 1 ORDER BY 2 DESC;
-- price 24.5 | no_need 19.1 | quality 16.6 | consolidation 15.9 | ai_alternative 14.7 | other 9.3
```

## Находка №2 — team-009

**Команда:** team-009 (`POST /api/chat`)

**Вопрос:** «Какая главная причина оттока в сегменте Large?» (то же на «…в сегменте Mid?»)

**Ответ системы:**
> «Главная причина оттока в сегменте Large — цена, на неё приходится **24,6%** случаев. При этом структурные причины (ai_alternative, no_need, consolidation) в сумме составляют **49,1%** оттока.»
> (для Mid — ровно те же числа: «цена 24,6%», «49,1%».)

**Что не так:** Пропущен фильтр по сегменту — система вернула **глобальные** числа (24.6% и 49.1% — это распределение по всей базе), одинаковые для Large и Mid. Реальные доли по сегментам отличаются, а одинаковый ответ для двух разных сегментов доказывает отсутствие среза.

**Как должно быть:**
- Large: цена **25.0%**, сумма ai+no_need+consolidation = **47.7%**.
- Mid: цена **24.9%**, сумма = **48.1%**.
- (Глобально: 24.6% и 49.1%.)

**Проверка (DuckDB):**
```sql
SELECT c.segment,
  round(100.0*count(*) FILTER(WHERE cr.primary_reason='price')/count(*),1) price_pct,
  round(100.0*count(*) FILTER(WHERE cr.primary_reason IN('ai_alternative','no_need','consolidation'))/count(*),1) struct_pct
FROM churn_reasons cr JOIN customers c USING(customer_id) GROUP BY 1;
-- Large 25.0/47.7 | Mid 24.9/48.1 | SMB 24.5/49.6 ; глобал 24.6/49.1
```

## Находка №3 — team-009

**Команда:** team-009 (`POST /api/chat`)

**Вопрос:** «Какой NPS по каждому сегменту?»

**Ответ системы:**
> «**Средняя оценка NPS составляет 7,9.** Чаще всего клиенты относятся к категории промоутеров (46%)…» (разбивки по сегментам нет.)

**Что не так:** Два дефекта. (1) Пропущен фильтр по сегменту — разбивки по сегментам нет вообще, хотя спросили именно её. (2) Неверная метрика: «7,9» — это средний балл ответов (avg score 0–10), а не NPS. NPS = %промоутеров − %детракторов = **24.8**, и подавать 7.9 как «NPS» неверно.

**Как должно быть:** NPS по сегментам: **Large 45.3 · Mid 23.5 · SMB 22.1** (разброс огромный — у Large вдвое выше). Общий NPS = 24.8 (а 7.9 — это средний балл, иная метрика).

**Проверка (DuckDB):**
```sql
SELECT c.segment, round(100.0*(count(*) FILTER(WHERE n.score>=9)-count(*) FILTER(WHERE n.score<=6))/count(*),1) nps
FROM nps_responses n JOIN customers c USING(customer_id) GROUP BY 1;
-- Large 45.3 | Mid 23.5 | SMB 22.1 ; общий NPS 24.8, avg_score 7.9
```

## Находка №5 — team-009

**Команда:** team-009 (`POST /api/chat`)

**Вопрос:** «Как изменилась доля спящих (неактивных) клиентов по годам?»

**Ответ системы:**
> «Точное изменение доли спящих клиентов по годам не представлено, но их количество — **61 145** — указывает на существенную величину. Ключевые факты: Количество спящих клиентов: **61 145**. Общее количество клиентов: **608 920**.»

**Что не так:** Грубая ошибка гранулярности (неверный факт). «608 920» — это число **помесячных записей активности**, а не клиентов; «61 145» — число **месяцев в статусе dormant**, а не уникальных спящих клиентов. Реально клиентов **~24 926 уникальных** (всего в витрине 25 000). Заодно не дан запрошенный тренд по годам, хотя он есть.

**Как должно быть:** Доля неактивных (dormant+churning) на конец года: **2023 — 18.4% → 2024 — 22.8% → 2025 — 32.5%** (растёт — признак реального «тихого» оттока). Уникальных клиентов ~24 926, а не 608 920.

**Проверка (DuckDB):**
```sql
SELECT count(*) rows, count(distinct customer_id) uniq,
  count(*) FILTER(WHERE status='dormant') dormant_rows FROM customer_activity_monthly;
-- rows 608920 | uniq 24926 | dormant_rows 61145  ← их «клиенты/спящие» = это СТРОКИ, не клиенты
SELECT year(month) yr, round(100.0*count(*) FILTER(WHERE status IN('dormant','churning'))/count(*),1) pct
FROM customer_activity_monthly WHERE month(month)=12 GROUP BY 1 ORDER BY 1;  -- 18.4 / 22.8 / 32.5
```
