## №1 — месяц с минимальным оттоком 2025
**Вопрос:** «В каком месяце 2025 года был самый низкий отток клиентов?»
**Ответ:** «Ответить нельзя — извлечённые данные содержат только годовую агрегацию (churn_rate за 2025 = 11,0%), но не помесячную разбивку.»
**Что не так:** Ложный отказ — помесячный отток выводится напрямую из даты ухода (`churn_date` имеет точность до месяца). Данные есть.
**Как должно быть:** Помесячно за 2025 отток считается (янв 177 … ноя 141, дек 25). При этом декабрь неполон (цензуринг) — корректный ответ либо назвать ноябрь (минимум среди полных), либо пометить декабрь как обрезанный; но заявлять «помесячных данных нет» неверно.
**Проверка:** `SELECT strftime(churn_date,'%Y-%m') m, count(*) FROM customers WHERE year(churn_date)=2025 GROUP BY 1 ORDER BY 1;`

## №2 — доля GMV по сегменту Large
**Вопрос:** «Какая доля GMV приходится на сегмент Large?»
**Ответ:** «Недостаточно данных в витрине. Отсутствующие поля: unit_economics_monthly_clean.gmv. Planner-gate отбил вопрос до SQL.»
**Что не так:** Ложный отказ — система искала GMV не в той таблице (в юнит-экономике GMV нет, но он есть в заказах). GMV по сегментам считается через заказы + сегмент клиента.
**Как должно быть:** Large = **25.3%** GMV (Mid 39.6%, SMB 35.2%).
**Проверка:** `SELECT c.segment, round(100.0*sum(o.gmv)/sum(sum(o.gmv)) OVER(),1) FROM orders o JOIN customers c USING(customer_id) GROUP BY 1;`

## №3 — доля детракторов/промоутеров
**Вопрос:** «Какая доля детракторов и промоутеров в NPS?»
**Ответ:** «Extraction-агент предоставил только итоговый NPS (35.42 за Q4 2025)… для долей данных нет.»
**Что не так:** Ложный отказ — доли считаются напрямую из баллов опросов.
**Как должно быть:** За 2025: промоутеры 47%, нейтралы 40%, детракторы 13% (за всё время 46% / 33% / 21.2%).
**Проверка:** `SELECT round(100.0*count(*) FILTER(WHERE score>=9)/count(*),1), round(100.0*count(*) FILTER(WHERE score<=6)/count(*),1) FROM nps_responses WHERE year(response_date)=2025;` → 47 / 13

## №4 — средний чек по сегментам
**Вопрос:** «Какой средний чек по сегментам?»
**Ответ:** «Недостаточно данных в витрине. Отсутствующие поля: orders_clean.segment. Planner-gate отбил вопрос до SQL.»
**Что не так:** Ложный отказ — сегмент берётся из таблицы клиентов джойном к заказам; средний чек по сегментам считается.
**Как должно быть:** Считается через `orders ⋈ customers.segment` (данные есть).
**Проверка:** `SELECT c.segment, round(avg(o.gmv),0) FROM orders o JOIN customers c USING(customer_id) GROUP BY 1;`

## №5 — доля спящих по годам
**Вопрос:** «Как менялась доля спящих клиентов по годам?»
**Ответ:** «Доля спящих (dormant) растёт: 33.8% (2023) → 45.9% (2024) → 51.5% (2025).»
**Что не так:** Неверные числа — не воспроизводятся ни по одному определению. dormant-only по годам = 7.2% / 11.0% / 12.0%; неактивные (dormant+churning) на конец года = 18.4% / 22.8% / 32.5%. Значения 33.8/45.9/51.5 завышены (максимум любой метрики ≈ 28–32%).
**Как должно быть:** dormant-only ≈ 12% (2025); неактивные (спящие+уходящие) 18.4% → 22.8% → 32.5%.
**Проверка:** `SELECT year(month) yr, round(100.0*count(*) FILTER(WHERE status IN('dormant','churning'))/count(*),1) FROM customer_activity_monthly WHERE month(month)=12 GROUP BY 1;` → 18.4 / 22.8 / 32.5

## №6 — клиенты по топ-5 городам
**Вопрос:** «Сколько клиентов в топ-5 городах?»
**Ответ:** «По какому критерию считать „лучший клиент“ — GMV или число заказов?… лидер 7535: 75.42 млн ₽…»
**Что не так:** Ответ не на вопрос — выдан текст про «лучшего клиента» по GMV вместо разбивки клиентов по городам. Город есть в данных (10 городов).
**Как должно быть:** Разбивка по `city` (10 городов) считается напрямую.
**Проверка:** `SELECT city, count(*) FROM customers GROUP BY 1 ORDER BY 2 DESC LIMIT 5;`

## №7 — средний чек
**Вопрос:** «Какой средний чек у клиента?»
**Ответ:** «Недостаточно данных… нет прямого поля „средний чек“…»
**Что не так:** Ложный отказ — средний чек (AOV) = среднее `orders.gmv` по заказам, считается напрямую.
**Как должно быть:** AOV ≈ 105 770 ₽ (avg gmv заказа).
**Проверка:** `SELECT round(avg(gmv),0) FROM orders;` → 105770
