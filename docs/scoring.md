# Оценка и автопроверки — AI South Hub 2026

## Итоговая формула
**45%** — автоматическая оценка LLM-судьёй по живому API.
**55%** — экспертная оценка жюри.

## Критерии (веса экспертной/общей рубрики)

| Критерий | Вес | Что это значит |
|---|---|---|
| Точность ответов | 30% | Цифры верны, метод расчёта корректен. |
| **Boundary discipline** | 15% | Система честно говорит «данных не хватает», когда это так, и не выдумывает. |
| Качество бизнес-интерпретации | 25% | Вывод на языке бизнеса, со стратегическими следствиями, а не сухие числа. |
| Архитектура агентов | 20% | Чёткое разделение Extractor/Analyst/Critic/Visualization, контракты обмена, самодиагностика. |
| Визуализация и UX | 10% | Правильный тип графика под задачу, удобство. |

## Что проверяет LLM-судья автоматически (по живому API)

1. **Контракт:** POST+JSON принимается; ответ — осмысленный русский UTF-8 ≥10 символов; вне-скоуп → «недостаточно данных».
2. **Ошибки (никаких 500):** пустое тело→400, кривой JSON→400, нет поля→400/422, левый путь→404; ошибки JSON-полем `error`/`detail`.
3. **Производительность:** простой <5 мин, аналитика <10 мин, 3 параллельных запроса, контекст в сессии.
4. **Робастность под стрессом (~20%):** входы 5000+ символов, спецсимволы/эмодзи/SQL/HTML, prompt injection, несовпадение типов — всё без 500.
5. **Бонусы:** `GET /health→200`, `/docs`|`/openapi.json`, CORS, SSE.

## Практические выводы для команды
- **Boundary discipline (15%) — дешёвый выигрыш:** надёжно детектить вне-скоуп вопросы и возвращать `insufficient_data: true`.
- **Critic (часть 20% архитектуры + влияет на точность):** строить конкретные проверки под категории ошибок (SQL, интерпретация, галлюцинации, нюансы).
- **Робастность ~20% автобалла** дают почти «бесплатно» аккуратной обработкой ошибок и валидацией ввода — закрыть в первую очередь.
- Готовить к защите **и** примеры отличных ответов, **и** примеры честных отказов — жюри тестит вживую.
