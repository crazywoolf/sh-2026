# AI South Hub 2026 — кейс «Meridian»

Контекст хакатона, собранный с https://dashboard.aisouthhack.ru/ (на 2026-06-11).

## Что нужно построить

**Мультиагентную аналитическую систему**, которая работает как data-аналитик для совета директоров компании Meridian. На вход — бизнес-вопрос на естественном языке (русский), на выход — обоснованный ответ с цифрами, методом расчёта, явными допущениями и честным признанием там, где данных не хватает.

Систему дёргает **LLM-судья по живому API** + смотрит жюри. Главное правило: **никогда не возвращать 500 и не выдумывать данные.** Если ответить нельзя — честно сказать «недостаточно данных».

## Требуемая архитектура — 4 агента

| Агент | Задача |
|---|---|
| **Extractor** | Интерпретирует вопрос → схема данных → SQL. Определяет, когда данных не хватает. |
| **Analyst** | Из таблиц выводит инсайты на языке бизнеса, ищет паттерны/аномалии, переводит цифры в стратегические выводы. |
| **Critic** | Проверяет интерпретацию вопроса, корректность среза данных, обоснованность выводов; ловит галлюцинации и подмены. **Ключевой компонент для оценки.** |
| **Visualization** | Выбирает тип графика под задачу (динамика / сравнение категорий / аномалии / структура). |

Архитектуру проектировать сразу под **три режима** (см. ниже), иначе переход с чата на отчёты дорогой.

## Три режима работы
1. **Conversational BI** — диалог в свободной форме, краткие ответы + визуал, удержание контекста между вопросами.
2. **Ad-hoc research** — система сама декомпозирует сложный вопрос и проводит мини-исследование.
3. **Scheduled reports** — автоматические дашборды здоровья бизнеса по расписанию.

## Оценка (45% LLM-судья / 55% жюри)

| Критерий | Вес |
|---|---|
| Точность ответов | 30% |
| **Boundary discipline** (честно сказать «данных не хватает») | 15% |
| Качество бизнес-интерпретации | 25% |
| Архитектура агентов | 20% |
| Визуализация и UX | 10% |

Плюс автопроверки робастности (~20% технической части): длинные входы, спецсимволы, prompt injection, конкурентные запросы.

## API-контракт (критично — по нему тестирует судья)

POST на один из путей (проверяются по порядку): `/api/chat` → `/api/v1/chat` → `/chat` → `/api/ask` → `/api/query`

**Запрос:** `{"message": "..."}` (или `query`, или `messages: [{role, content}]`); для диалога добавить `session_id`.

**Ответ (минимум):** `{"response": "вывод на языке бизнеса"}` (валидны также: `answer`, `message`, `content`, `text`).

**Рекомендованный расширенный ответ:**
```json
{
  "response": "Бизнес-вывод",
  "assumptions": ["допущения"],
  "trace": [
    {"agent": "extractor", "sql": "SELECT...", "rows": 120},
    {"agent": "analyst", "note": "наблюдение"},
    {"agent": "critic", "verdict": "approved"}
  ],
  "chart": {"type": "line", "x": "quarter", "y": "revenue"},
  "insufficient_data": false,
  "session_id": "abc123"
}
```
Нет ответа → `"insufficient_data": true`, **не выдумывать.**

**Обязательные правила robustness — никаких 500:**
- пустое тело → 400; кривой JSON → 400; нет поля вопроса → 400/422; несуществующий путь → 404
- ошибки — JSON-полем `error` или `detail`
- простой вопрос < 5 мин, аналитика < 10 мин, 3 параллельных запроса обслуживаются
- бонусы: `GET /health → 200`, `/docs` или `/openapi.json`, CORS, SSE (`text/event-stream`)

## Культурное требование Meridian
> Здесь «мне кажется» не работает. Работает «вот цифры, вот метод расчёта, вот допущения».
Не усреднять по разнородным группам, явно проговаривать допущения, отделять уверенные выводы от ограничений данных.

## Бизнес-контекст кейса
Meridian — B2B-маркетплейс услуг для среднего бизнеса. Выручка −4% г/г, отток клиентов 23%/год; клиенты уходят к специализированным AI-инструментам. Аналитический отдел отвечает на вопросы за 5–10 дней через тикеты — медленно. Нужно убрать эту задержку.
